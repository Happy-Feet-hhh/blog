## 📚 核心知识点

### 1️⃣ 计算机图形学的定义与本质
- **核心定义**：使用计算将**数字信息**转化为**感官刺激**（视觉/听觉/触觉等）
- **与计算机视觉的关系**：图形学是"合成"，视觉是"解析"，二者互为逆问题
- **为什么重视视觉**：约30%大脑用于视觉处理，眼睛是大脑的最高带宽输入端口
![](Pasted%20image%2020260418090431.png)
### 2️⃣ 历史演进与数据带宽

|时代|交互方式|信息带宽|局限性|
|---|---|---|---|
|1945 (ENIAC)|打孔卡片|~120 bytes/卡|输入输出效率极低，难以理解结果|
|1963 (Sketchpad)|光笔+示波器|首次实现图形交互|分辨率低，功能有限|
|2020 (8K显示器)|高分辨率屏幕|~95 MB/帧|视觉信息密度大幅提升|
|2020 (VR头显)|双目90Hz显示|~2.3 GB/s|实时渲染压力巨大|
#### 2.2 为什么视觉是核心通道？（神经科学依据）
🧠 大脑资源分配：
   • 约30%的大脑皮层专门用于视觉处理
   • 眼睛是大脑的"最高带宽输入端口"
   • 视觉信息处理速度远超听觉/触觉

💡 结论：若目标是将机器信息高效传递给人类，视觉是最优选择

#### 2.3 图形学的三层定义演进

📌 狭义定义（传统）：
   "使用计算机合成视觉信息"
   digital information → computation → visual information

📌 中义定义（扩展）：
   "使用计算机合成和操纵感官信息"
   • 视觉：图像、视频、3D渲染
   • 听觉：空间音频、3D音效合成
   • 触觉：力反馈、虚拟材质感知

📌 广义定义（本课程采用）：
   "使用计算将数字信息转化为感官刺激"
   computation → digital information → sensory stimuli
   
   延伸方向：
   • 3D打印：digital → physical matter
   • 数字制造：服装/食品/建筑的数字化生成
   • 医疗假体：扫描→建模→打印→临床应用

#### 2.4 图形学 vs 计算机视觉：互为逆问题
🔄 计算机图形学（合成）：
   数字模型 + 光照/材质/相机 → 生成图像/感官刺激

🔄 计算机视觉（解析）：
   真实世界图像/视频 → 提取数字信息/理解场景

💡 二者结合：增强现实、三维重建、神经渲染等前沿方向

### 5️⃣ 实践案例：立方体建模与渲染

#### 🎯 建模（Modeling）

既然计算机图形学得定义是使用计算将**数字信息**转化为**感官刺激**（视觉/听觉/触觉等），那么让我们想象一下，我们如何在计算机空间中**描述**和**展示**一个正方体。也就是如何渲染它。
![](Pasted%20image%2020260418091054.png)
当我们谈论渲染时，其实就是在谈论如何将数字描述，转化为一个具体的图像。

关于如何描述，即如何建模，可以有以下的情况

###### 🔹 几何设定
• 立方体尺寸：2×2×2（边长为2）
• 中心位置：原点 (0, 0, 0)
• 朝向：边平行于X/Y/Z轴（axis-aligned）

###### 🔹 顶点表示（Vertex List）
```tex
8个顶点，坐标为 (±1, ±1, ±1) 的所有组合：

A: ( 1,  1,  1)    E: ( 1,  1, -1)
B: (-1,  1,  1)    F: (-1,  1, -1)
C: ( 1, -1,  1)    G: ( 1, -1, -1)
D: (-1, -1,  1)    H: (-1, -1, -1)

💡 设计思想：
• 顶点列表可泛化到任意网格模型
• 每个顶点可扩展属性：法线、纹理坐标、颜色等
```

###### 🔹 边表示（Edge List）
```tex
12条边，用顶点对表示（确保仅一个坐标变化）：

// X方向边（Y,Z固定）
AB: A(1,1,1) ↔ B(-1,1,1)   [X:1→-1]
CD: C(1,-1,1) ↔ D(-1,-1,1) [X:1→-1]
EF: E(1,1,-1) ↔ F(-1,1,-1) [X:1→-1]
GH: G(1,-1,-1) ↔ H(-1,-1,-1) [X:1→-1]

// Y方向边（X,Z固定）
AC: A(1,1,1) ↔ C(1,-1,1)   [Y:1→-1]
BD: B(-1,1,1) ↔ D(-1,-1,1) [Y:1→-1]
EG: E(1,1,-1) ↔ G(1,-1,-1) [Y:1→-1]
FH: F(-1,1,-1) ↔ H(-1,-1,-1) [Y:1→-1]

// Z方向边（X,Y固定）
AE: A(1,1,1) ↔ E(1,1,-1)   [Z:1→-1]
BF: B(-1,1,1) ↔ F(-1,1,-1) [Z:1→-1]
CG: C(1,-1,1) ↔ G(1,-1,-1) [Z:1→-1]
DH: D(-1,-1,1) ↔ H(-1,-1,-1) [Z:1→-1]

💡 验证方法：
• 每条边两端点应恰好有一个坐标不同
• 不同坐标差值应为±2（边长为2）
```

所以我们必须保留3维信息，但是又要呈现2D的画面，该怎么办呢？一个基本的想法如下：
![](Pasted%20image%2020260418092531.png)
但这个只是一个想法，我们还没有具体的算法，不是吗？我们根本不知道如何编写程序来完成这种转换。
要完成这样的算法设计，我们必须回想自己的日常生活经历。当我们在生活中观察物体的时候，会发现有**近大远小**的的**透视**现象。
![](Pasted%20image%2020260418092845.png)
针孔相机模型（Pinhole Camera），如上图所示， 暗箱 + 小孔 + 成像平面（胶片/传感器）组成了针孔相机，我们为了简化理解，假设小孔位置就是相机中心，即假设 C = (0,0,0)（C为相机中心，亦为小孔中心）。

从这个模型，我们可以很直观的看到，为什么当物体远离人眼的时候，人会觉得物体变小，我们还是以这个针孔相机模型为例，不过这一次，我们从**侧面**来观察：
![](Pasted%20image%2020260418093405.png)
假设，上图中的树叶反射出了一道绿色的光线，然后通过小孔，照射到了film(胶片)或者image上，那么树叶点P对应的像素点q应该如何计算？

答案是相似三角形，从这个图中我们不好想象图形的u坐标(因为u坐标无论怎么变，我们在侧面观察不出来)，我们从纵坐标V来观察。
![](Pasted%20image%2020260418093719.png)
我们假设相机的尺寸为1，那么根据相似三角形的原理，有以下公式：
$$
\frac{V}{1} = \frac{Y}{Z}
$$
随着距离Z越来越大，p点离相机中心的c点的纵坐标之差越来越小，即v会越来越小，同理，u也会越来越小(不过这个要我们从俯视角去观察)。

###### 🔹 投影公式总结
```
给定3D点 P=(X,Y,Z) 和相机位置 C=(Cx,Cy,Cz)：

1️⃣ 坐标变换（世界→相机坐标系）：
   x = X - Cx
   y = Y - Cy  
   z = Z - Cz

2️⃣ 透视除法（3D→2D）：
   u = x / z
   v = y / z

⚠️ 注意：
• 要求 z < 0（点在相机前方，成像平面在负Z）
• 若 z > 0，点在相机后方，投影无意义
• 实际系统会用齐次坐标处理 z=0 边界情况
```

$$
u = x / z
$$
$$
v = y / z
$$

#### 🎯 光栅化（Rasterization）
虽然我们之前解释了如何建模和渲染一个四边形，但一个新的问题出现了，我们怎么去画线段呢？
显示器由RGB子像素阵列组成，每个像素是发光单元，非"小方块"
![](Pasted%20image%2020260418095354.png)
对于一条线段，我们只需要绘制它穿过的像素点即可，但是如何缺点点亮那一个，其实没有最“正确”的方法，因为很多采样点比较牵强，我们不必绘制。现代一版采用中心菱形来采样，如下图所示。
![](Pasted%20image%2020260418095436.png)

当明确了采样的规则后，我们如何完成算法来保证每一个需要采样的点都被计算呢？笨拙的的算法是直接扫描屏幕的每一个像素点(从左往右，从上往下)，找到穿过中心菱形的点，然后点亮。

但是这太浪费了，算法的时间复杂度为O(n^2)。并且当你的屏幕分辨率变高的时候，会增加更多的时间复杂度。

所以我们可以采取斜率计算的方式：
![](Pasted%20image%2020260418100239.png)
我们只用遍历u1到u2的范围，然后每一次迭代，都对v坐标增加一次斜率值。然后绘制点(u, round(v))，这里对v取整。这样就可以在时间复杂度O(n)的情况下完成计算。

这样，我们仅仅通过这一章导论的学习，就完成了从3D物体的数字信息到生成2维视觉图像的完整流程。
![](Pasted%20image%2020260418100448.png)
##### 总结
- **像素表示**：图像 = 2D网格，每个单元格存储颜色值
- **线条光栅化规则**：
  - 简单规则：线穿过像素即点亮
  - Diamond Rule（现代GPU采用）：线穿过像素中心菱形区域才点亮
- **增量式算法**（斜率 0<s<1 时）：
```python
v = v1
for u in range(u1, u2+1):
    v += s  # s = (v2-v1)/(u2-u1)
    draw(u, round(v))
```
- 复杂度：从 O(n²) 优化到 O(n)

1. **手动计算透视投影**
   - 给定相机位置 `C=(2,3,5)`，计算立方体8个顶点的2D投影坐标
   - 验证公式：`先减去相机坐标 → u=x/z, v=y/z`
   - ✅ 参考答案：A(1/4,1/2), B(3/4,1/2), C(1/4,1), D(3/4,1)...

```rust
/// 三维向量结构体
#[derive(Debug, Copy, Clone)]
struct Vec3 {
    x: f32,
    y: f32,
    z: f32,
}
  
// 立方体的坐标
const VERTICES: [Vec3; 8] = [
    Vec3{ x: 1., y: 1., z: 1. },       // A: ( 1,  1,  1)
    Vec3{ x: -1., y: 1., z: 1. },      // B: (-1,  1,  1)
    Vec3{ x: 1., y: -1., z: 1. },      // C: ( 1, -1,  1)
    Vec3{ x: -1., y: -1., z: 1. },     // D: (-1, -1,  1)
    Vec3{ x: 1., y: 1., z: -1. },      // E: ( 1,  1, -1)
    Vec3{ x: -1., y: 1., z: -1. },     // F: (-1,  1, -1)
    Vec3{ x: 1., y: -1., z: -1. },     // G: ( 1, -1, -1)
    Vec3{ x: -1., y: -1., z: -1. },    // H: (-1, -1, -1)
];

// 立方体12条边（顶点索引对）
const CUBE_EDGES: [(usize, usize); 12] = [
    (0,1), (1,3), (3,2), (2,0), // 正面
    (4,5), (5,7), (7,6), (6,4), // 背面
    (0,4), (1,5), (2,6), (3,7), // 连接面
];
  
// 相机的坐标
const CAMERA_LOCATION: Vec3 = Vec3{ x: 2., y: 3., z: 5. }; // c = (2, 3, 5)
  
// 获取uv坐标的复制函数
fn from_xyz_get_uv(vertex: Vec3) -> (f32, f32) {
    // 得到以相机位置为原点的目标顶点的x和y坐标, 以及z坐标(距离)，即转成「以相机为原点」的相对坐标
    let x = vertex.x - CAMERA_LOCATION.x;
    let y = vertex.y - CAMERA_LOCATION.y;
    let z = vertex.z - CAMERA_LOCATION.z;

    // 透视除法：把3D坐标投射到2D平面（简单版透视投影）模拟人眼的 “近大远小”
    let u = x / z;
    let v = y / z;

    (u, v)
}
```
根据我们之前对立方体的顶点和边关系的描述。以上代码声明了对应的数据结构，然后通过`from_xyz_get_uv()`函数得到了立方体的每一个顶点的投影坐标。

2. **实现增量式线条光栅化**
   - 编写函数处理不同斜率情况（|s|<1 和 |s|>1）
   - 理解为什么需要交换u/v遍历顺序

根据题目写出以下的代码：
```rust
struct FrameBuffer {
    width: usize,
    height: usize,
    // 像素格式：(R, G, B)，0~255
    pixels: Vec<(u8, u8, u8)>,
}

impl FrameBuffer {
    /// 创建空白帧缓冲
    fn new(width: usize, height: usize) -> Self {
        let pixels = vec![(0, 0, 0); width * height]; // 黑色背景
        FrameBuffer { width, height, pixels }
    }

    /// 清空画布为指定颜色
    fn clear(&mut self, r: u8, g: u8, b: u8) {
        for p in &mut self.pixels {
            *p = (r, g, b);
        }
    }

    /// 画一个点（安全边界检查）
    fn draw_pixel(&mut self, x: i32, y: i32, r: u8, g: u8, b: u8) {
        if x >= 0 && x < self.width as i32 && y >= 0 && y < self.height as i32 {
            let idx = y as usize * self.width + x as usize;
            self.pixels[idx] = (r, g, b);
        }
    }

    /// Bresenham 直线算法：纯光栅化画线（核心）
    fn draw_line(&mut self, mut x0: i32, mut y0: i32, x1: i32, y1: i32, r: u8, g: u8, b: u8) {
        let dx = (x1 - x0).abs();
        let dy = -(y1 - y0).abs();
        let sx = if x0 < x1 { 1 } else { -1 };
        let sy = if y0 < y1 { 1 } else { -1 };

        let mut err = dx + dy;

        loop {
            self.draw_pixel(x0, y0, r, g, b);
            if x0 == x1 && y0 == y1 { break; }
            let e2 = 2 * err;
            if e2 >= dy {
                err += dy;
                x0 += sx;
            }
            if e2 <= dx {
                err += dx;
                y0 += sy;
            }
        }
    }


    /// 输出为 PPM 图片（纯文本格式，系统可直接打开）
    fn save_ppm(&self, path: &str) -> std::io::Result<()> {
        use std::fs::File;
        use std::io::Write;

        let mut file = File::create(path)?;
        writeln!(file, "P3")?; // 2. PPM格式标识：P3 = 纯文本RGB格式
        writeln!(file, "{} {}", self.width, self.height)?;  // 3. 画布宽高（像素数）
        writeln!(file, "255")?; // 4. 颜色分量最大值（0~255，固定值）
  
        // 5. 遍历所有像素，写入RGB值
        for &(r, g, b) in &self.pixels {
            writeln!(file, "{} {} {}", r, g, b)?;
        }
        Ok(())
    }
}
```

```rust
/// Bresenham 直线算法：纯光栅化画线（核心）
    ///
    /// 原理：
    ///   屏幕是离散的像素网格，"画一条直线"本质上是决定哪些像素该被点亮。
    ///   Bresenham 算法只用整数加减法就能逼近理想直线，是经典的光栅化算法。
    ///
    ///   核心思想是维护一个"误差值" err：
    ///   - err 衡量当前像素点 (x0, y0) 偏离理想直线有多远
    ///   - 每走一步，检查 err 来决定下一步是横着走、竖着走、还是对角走
    ///   - 这样选出的像素序列最接近数学上的直线
    ///
    /// 参数：
    ///   (x0, y0) 起点，(x1, y1) 终点（x0/y0 被 mut 修饰，算法执行中会不断移动）
    ///   (r, g, b) 线条颜色
    ///
    /// 举例（从 (0,0) 画到 (5,2)）：
    ///
    ///   y ↑
    ///     |  ·  ·  ·  ·  ·  #
    ///     |  ·  ·  #  ·  ·  ·
    ///     |  #  ·  ·  #  ·  ·
    ///     |  ·  #  ·  ·  #  ·
    ///     |  ·  ·  #  ·  ·  ·
    ///     +——————————————→ x
    ///
    ///   算法从起点 (0,0) 出发，每步向右走一格，
    ///   根据误差值决定是否同时向上走一格，最终到达 (5,2)。
    fn draw_line(&mut self, mut x0: i32, mut y0: i32, x1: i32, y1: i32, r: u8, g: u8, b: u8) {
        // ── 第一阶段：计算直线的基本参数 ──
  
        // dx：x 方向上的绝对距离（水平跨度，永远 ≥ 0）
        let dx = (x1 - x0).abs();
        // dy：y 方向上的绝对距离，取负值（原因见下方 err 说明）
        let dy = -(y1 - y0).abs();
        // sx：x 方向步进符号。起点在终点左边 → 向右走(+1)，否则向左走(-1)
        let sx = if x0 < x1 { 1 } else { -1 };
        // sy：y 方向步进符号。起点在终点下边 → 向上走(+1)，否则向下走(-1)
        let sy = if y0 < y1 { 1 } else { -1 };

        // err：初始误差 = dx + dy（因为 dy 是负数，所以 err = dx - |dy|）
        // 含义：直线斜率越接近 0（越水平），初始 err 越大，x 方向主导行走
        //       直线斜率越接近 ∞（越垂直），初始 err 越小，y 方向主导行走
        let mut err = dx + dy;

        // ── 第二阶段：逐步像素化直线 ──
        loop {
            // 1. 在当前像素位置画一个点
            self.draw_pixel(x0, y0, r, g, b);
            // 2. 如果已经走到终点，结束
            if x0 == x1 && y0 == y1 { break; }
  
            // 3. 将误差放大 2 倍用于判断（避免浮点运算，乘 2 等价于比较阈值减半）
            let e2 = 2 * err;

            // 4. 判断是否需要水平移动：
            //    e2 >= dy  等价于  2·err >= -|dy|  等价于  err >= -|dy|/2
            //    含义：当前偏移还没大到需要在 y 方向修正，先在 x 方向走一步
            //    直观理解：如果直线偏水平（dx > |dy|），这个条件大部分时候成立
            if e2 >= dy {
                err += dy;    // 累加 y 方向的误差（dy 是负数，所以 err 减小）
                x0 += sx;     // x 前进一步
            }

            // 5. 判断是否需要垂直移动：
            //    e2 <= dx  等价于  2·err <= dx  等价于  err <= dx/2
            //    含义：误差已经累积到需要在 y 方向走一步来追赶理想直线
            //    直观理解：如果直线偏垂直（|dy| > dx），这个条件大部分时候成立
            if e2 <= dx {
                err += dx;    // 累加 x 方向的误差（dx 是正数，所以 err 增大）
                y0 += sy;     // y 前进一步
            }

            // 注意：两个 if 可以同时成立 → 对角移动（同时 x 和 y 各走一步）
        }

    }
```
最后在`main`函数中完成绘制，并输出ppm文件：
```rust
fn main() -> std::io::Result<()> {
    // 1. 创建 800x600 帧缓冲
    let mut fb = FrameBuffer::new(800, 600);
    fb.clear(110, 110, 110); // 灰色背景

    // 2. 遍历立方体12条边，逐边画线
    for &(i, j) in &CUBE_EDGES {
        let v0 = VERTICES[i];   // 取边的起点3D坐标
        let v1 = VERTICES[j];   // 取边的终点3D坐标
        // 3D→UV（透视投影）
        let uv0 = from_xyz_get_uv(v0);
        let uv1 = from_xyz_get_uv(v1);

        // UV→像素坐标
        let (x0, y0) = uv_to_screen(uv0, fb.width, fb.height);
        let (x1, y1) = uv_to_screen(uv1, fb.width, fb.height);

        // 画白色直线
        fb.draw_line(x0, y0, x1, y1, 255, 255, 255); // 白色线
    }

    // 3. 保存为 PPM 图片
    fb.save_ppm("cube.ppm")?;
    println!("立方体渲染完成！已生成 cube.ppm 文件");
  
    Ok(())
}
```

## 1. 创建渲染器骨架

### 1.1 骨架代码

在上一节，我们搭建好了WebGPU的开发环境，这一节，我们开始编写具体的代码。由于本项目是一个实战驱动的项目，所以不会一上来就介绍全部的概念，而是在实战的过程中一点一点的穿插。

首先，我们要实现的app是一个渲染器Renderer，所以我们在`src/main.ts`中编写如下的代码：
```typescript
// src/main.ts
class Renderer {
  private context!: GPUCanvasContext;

  public async initialize() {
    // 在初始化方法中准备画布canvas
    const canvas: HTMLCanvasElement = document.getElementById("canvas") as HTMLCanvasElement;
    // 调用画布的核心方法，请求创建 WebGPU 渲染上下文，渲染前必须通过canvas.getContext获取上下文
    this.context = canvas.getContext('webgpu')!;
  
    if (!this.context) {
      alert('WebGPU 在该环境不受支持!');
      return;
    }
  }
  
  public draw() {
  }

}

const renderer = new Renderer();

// 由于渲染器Renderer的初始化是异步的，所以我们直接调用then方法
// 当它初始化完成后立马调用draw()
renderer.initialize()
  .then(() => renderer.draw())
```

以上的代码只做了一件事情，那就是创建了渲染器Renderer这个类型，然后再里面声明了一个WebGPU的上下文私有变量：`context: GPUCanvasContext`。我们的程序只有获取到了canvas的上下文之后才能在canvas上渲染画面。顺带提一嘴，在WGPU中这应该就是Surface。

而在初始化的代码里面，我们只是获取了WebGPU的canvas上下文，然后做了一个检测，最后使用函数式的链式调用方式，处理异步的初始化函数，一旦初始化函数完成，即可调用draw函数绘制画面。

所以有了这样的代码之后，使用npm run dev，打开浏览器，如果浏览器没有报alert的错误，那么说明WebGPU是可以支持的。

有人可能会疑惑`this.context = canvas.getContext('webgpu')!;`那里不是已经断言非空了吗，为啥还有用if判断，因为不断言ts报错....

### 1.2 Navigator
每个现代浏览器都提供了一个 `Navigator` 对象，其属性包含了有关浏览器功能的信息。例如， `clipboard` 属性可以读取系统剪贴板， `connection` 属性可以检查网络连接， `deviceMemory` 属性可以识别可用的内存有多少。

如果浏览器支持WebGPU， `Navigator` 对象将有一个名为 `gpu` 的属性。以下代码展示了应用程序如何检查这个属性：
```typescript
 // Check if the browser supports WebGPU
 if (!navigator.gpu) {
    throw new Error("WebGPU not supported.");
 }
```
当然，也可以向我们那样，用`canvas`的`getContext('webgpu')`来判断。

`gpu` 属性提供了两个重要的方法：

- `requestAdapter()` ——返回一个`Promise`，该`Promise`在履行时会返回一个 `GPUAdapter` 。
- `getPreferredCanvasFormat()` ——返回一个字符串，标识浏览器画布中图形的最佳格式。

这些方法在WebGPU应用程序中扮演着重要的角色。我将在本章后面讨论这两个方法。

## 2. 完善初始化initialize方法

### 2.1 完整代码预览

在完成了基本的代码骨架后，我们继续初始化方法的完善，毕竟现在的代码只是一个能检测当前浏览器是否支持WebGPU的代码。而WebGPU本身的代码呢？马上就来。

我们在`src/main.ts`中键入如下的代码：
```typescript
// src/main.ts
class Renderer {
  private context!: GPUCanvasContext;
  private device!: GPUDevice;
  
  public async initialize() {
    // 在初始化方法中准备画布canvas
    const canvas: HTMLCanvasElement = document.getElementById("canvas") as HTMLCanvasElement;
    // 调用画布的核心方法，请求创建 WebGPU 渲染上下文，渲染前必须通过canvas.getContext获取上下文
    // ! 为Typescript的非空类型断言
    this.context = canvas.getContext('webgpu')!;
  
    if (!this.context) {
      alert('WebGPU 在该环境不受支持!');
      return;
    }

    // 同样requestAdapter返回的是Promise<GPUAdapter | null>，所以需要判空
    const adapter = await navigator.gpu.requestAdapter({
      powerPreference: "low-power"
    });

    if (!adapter) {
      alert("无法找到合适的适配器(显卡)")
    }

    // 逻辑设备，device是用户(我们)与adapter之间通信的接口
    this.device = await adapter?.requestDevice()!;

    // 当前浏览器 / 系统最推荐、兼容性最好的画布像素格式，通常返回 rgba8unorm 或 bgra8unorm
    // 作用：保证渲染颜色正确、不报错、性能最优，类似WPGU设置Surface
    this.context.configure({
      device: this.device,
      format: navigator.gpu.getPreferredCanvasFormat()
    });
  }

  

  public draw() {

  }

}

// ... 其余代码
```

在这里，我们添加了一个新的私有属性，`device!: GPUDevice`， 它是什么我们先按下不表，毕竟还没有到它登场的时候，我们先看`initialize()`函数的更新。

### 2.2 适配器 GPUAdapter

我们首先做了如下的操作：
```typescript
	// 同样requestAdapter返回的是Promise<GPUAdapter | null>，所以需要判空
    const adapter = await navigator.gpu.requestAdapter({
      powerPreference: "low-power"
    });

    if (!adapter) {
      alert("无法找到合适的适配器(显卡)")
    }
```
望文生义，至少从if判断中我们还是可以判断出，`adapter`指的是我们的物理显卡。那么WebGPU的标准如何定义`Adapter`的呢？我们看下[MDN的文档](https://developer.mozilla.org/zh-CN/docs/Web/API/GPUAdapter)：
![[Pasted image 20260413215639.png]]
在[learn-wgpu](https://jinleili.github.io/learn-wgpu-zh/beginner/tutorial2-surface/)中，也有详细的解释：

> **适配器**（Adapter）是指向 WebGPU API 实现的实例，一个系统上往往存在多个 WebGPU API 实现实例。也就是说，**适配器**是固定在特定图形后端的。假如你使用的是 Windows 且有 2 个显卡（集成显卡 + 独立显卡），则至少有 4 个**适配器**可供使用，分别有 2 个固定在 Vulkan 和 DirectX 后端。我们可以用它获取关联显卡的信息，例如显卡名称与其所适配到的后端图形驱动等。

基本上你可以把它看成显卡这个硬件，不那么讲究的话。至于会什么`requestAdapter()`返回的是`Promise<GPUAdapter | null>`，是因为客户端的浏览器支持WebGPU，并不意味着客户端的硬件支持WebGPU渲染和计算。为了确保这些功能可用，应用程序需要通过调用 `Navigator` 的 `gpu` 属性的 `requestAdapter` 方法来获取一个 `GPUAdapter` 。

如果 Promise 被履行， `adapter` 将被设置为一个 `GPUAdapter` 对象。如果没有， `adapter` 变量将被设置为 `null` 。

> 以下代码展示了如何通过Adapter获取硬件信息：
> ```typescript
> // 使用info属性可以获取硬件的属性
    const info = await adapter?.info;
    console.log(info?.vendor);      // 生产GPU的公司（例如nvidia、amd、intel）
    console.log(info?.architecture);  // 设备的架构
    console.log(info?.device);     // 设备的名称
    console.log(info?.description);  // 设备的描述
> ```

为了获取更多信息，应用程序可以访问 GPUAdapter 的两个有用属性：

- features —— 一个 GPUSupportedFeatures 对象，标识GPU支持的功能。
- limits —— 一个 GPUSupportedLimits 对象，标识GPU的最大/最小运行参数。

当应用程序需要了解GPU的能力时， features 属性会很有帮助。表2.2列出了可以检查的不同功能，并提供了每个功能的描述。

一个示例将展示如何访问这些功能。假设一个应用程序需要知道客户端的GPU是否支持深度裁剪控制，它可以使用以下代码：

```typescript
// 检查客户端是否支持深度裁剪控制功能
 if (adapter.features.has("depth-clip-control")) {
    console.log("Supported");
 } else {
    console.log("Unsupported");
 }
```

如果你想打印系统上所有可用的功能，你可以使用以下代码：

```typescript
// Display all of the supported features
 adapter.features.forEach((value) => {
    console.log(value);
 });
```

当应用程序需要了解某项功能的最大参数时， limits 属性会很有帮助。例如，以下代码确定了应用程序可以创建的顶点缓冲区的最大数量：

```typescript
const maxBuffers = adapter.limits.maxVertexBuffers;
```

`powerPreference` 枚举有两个可选项：`low-power` 和 `high-performance`。 `low-power` 对应偏向于高电池续航的适配器（如集成显卡上的 WebGPU 实现实例），`high-performance` 对应高功耗高性能的适配器（如独立显卡上的 WebGPU 实现实例）。一旦不存在符合 `high-performance` 选项的适配器，WebGPU 就会选择 `low-power`。

![[Pasted image 20260413220220.png]]


所以这里我们创建了一个低功耗的`adpater`。可能会对应你的集成显卡，谁知道呢，总之我们在初始的阶段不需要高性能的`adpater`。

### 2.3 逻辑设备 GPUDevice

在请求完硬件适配器`adapter`之后，我们还需要请求逻辑设备`GPUDevice`。

```typescript
// 逻辑设备，device是用户(我们)与adapter之间通信的接口
this.device = await adapter?.requestDevice()!;
```

有人可能会疑惑，我都拿到显卡了，为啥不能通过`adapter`直接去绘制？

一来是为了安全：**网页绝对不能直接操控物理硬件，必须通过浏览器授权的「逻辑设备」间接操作！**。

二来也是为了`物理 ↔ 逻辑 分离`，**一个 Adapter（物理显卡）可以创建 N 个 Device（逻辑设备）**
    
    同一个显卡，可以给：
    
    - 标签页 A 分配一个 Device
    - 标签页 B 分配一个独立 Device
    - 你的 WebGPU 程序 分配一个 Device
    

它们**完全隔离、互不干扰**。如果直接用 Adapter，一个网页崩溃 → 整张显卡挂掉 → 所有网页黑屏。

所以`GPUDevice` 作为与客户端GPU的逻辑连接，在WebGPU开发中扮演着核心角色。如表2.3所列，其方法创建的对象使得执行计算任务和图形操作成为可能。

### 2.4 canvas context 画布的上下文对象

在初始化代码的最后我们使用了如下代码：
```typescript
// 当前浏览器 / 系统最推荐、兼容性最好的画布像素格式，通常返回 rgba8unorm 或 bgra8unorm
// 作用：保证渲染颜色正确、不报错、性能最优，类似WPGU设置Surface
this.context.configure({
  device: this.device,
  format: navigator.gpu.getPreferredCanvasFormat()
});
```
前面我们知道，context是浏览器画布的上下文，我们要在画布上渲染东西就必须拿到它，现在我们要开始理解，为何上下文这么重要？

因为`canvas context`定义了一系列`texture`纹理的属性。纹理可以简单理解为渲染到画布上面的图像(暂时这么理解)。

画布上下文属性如下：

|属性|描述|
|---|---|
|device|控制画布的 GPUDevice|
|usage|标识获取的纹理将如何使用|
|alphaMode|控制透明度和不透明度|
|colorSpace|获取的纹理的颜色格式|
|format|获取的纹理的像素格式|
|viewFormats|获取的纹理的格式数组|
这些属性中的大多数与纹理有关，我们将在第6章中详细讨论。**前两个属性是最重要的**。第一个属性 `device` ，应该设置为前面讨论的 `GPUDevice` 。这是通过 `GPUAdapter` 获取的，代表客户端系统上的GPU。

第二个属性 `usage` 需要更多的解释。不同的计算机支持不同的纹理显示格式。 `usage` 字段告诉WebGPU应该使用什么纹理格式来呈现画布的内容。

你可能事先不知道该使用什么纹理格式。幸运的是，你可以使用 `Navigator` 的 `gpu` 属性的 `getPreferredCanvasFormat` 方法。这也是为什么我们在配置上下文的时候使用了：

```typescript
this.context.configure({
  device: this.device,   // 告诉上下文我们的设备是什么
  format: navigator.gpu.getPreferredCanvasFormat()   // 获取纹理的最佳格式
});
```

它检查用户系统并返回一个标识纹理像素最佳格式的字符串。它返回以下两个值之一：

- `rgba8unorm` ——每个像素包含四个8位值，分别标识红色、绿色、蓝色和透明度，每个值的范围从0到255。
- `bgra8unorm` ——每个像素包含四个8位值，分别标识蓝色、绿色、红色和透明度，每个值的范围从0到255。

## 3. 渲染图像与GPUCommandEncoder

```typescript
    // WebGPU API 中的 GPUCommandEncoder 接口是所有命令编码器的基础接口。
    // 它用于创建命令缓冲区
    const commandEncoder = this.device.createCommandEncoder();
    
    // WebGPU API 的 GPURenderPassDescriptor 接口
    // 用于描述一个渲染通道。
    // 它用于创建渲染通道编码器。
    const renderPassDescriptor: GPURenderPassDescriptor = {
      // WebGPU API 的 GPURenderPassDescriptor 接口的 colorAttachments 属性是一个颜色附件数组。
      // 它用于描述渲染通道的颜色附件。
      // 它用于创建渲染通道编码器。
      colorAttachments: [
        {
          // clearColor 用于描述纹理将被清除为的颜色。
          clearValue: { r: 0.8, g: 0.8, b: 0.8, a: 1.0 },
          // loadOp 用于描述纹理将如何加载，在本例中，我们正在清空纹理
          loadOp: "clear",
          // toreOp 用于描述纹理的存储方式。在这种情况下，我们正在存储该纹理。
          storeOp: "store",
          // 视图用于描述将被渲染到的纹理。
          view: this.context.getCurrentTexture().createView()
        }
      ]
    };

  

    // beginRenderPass 用于创建渲染通道编码器。
    // 每个渲染通道调用一次该方法。
    // passEncoder 用于为渲染通道编码指令。
    const passEncoder = commandEncoder.beginRenderPass(renderPassDescriptor);

    // DRAW HERE 以后渲染的代码在这里填充

    // END DRAW

    // endPass 用于结束渲染通道编码器。
    passEncoder.end();
    
    // submit 用于向图形处理器（GPU）提交命令缓冲区。
    // commandEncoder.finish () 用于创建命令缓冲区。
    this.device.queue.submit([commandEncoder.finish()]);

  }
```

### 3.1 WebGPU渲染流程原理解释

#### 整体背景

要谈到`CommandEncoder`，就要说到整体的历史，传统的 WebGL 基于旧版 OpenGL 设计，采用 “即时模式”：**CPU 每调用一个渲染函数，就会立刻把指令发送给 GPU 执行，CPU 需要等待 GPU 的响应**，无法充分利用两者的并行性。

而 WebGPU 是基于 Vulkan、Metal、DirectX 12 这些现代底层 GPU API 设计的，采用 “显式命令模式”：**CPU 先把所有要 GPU 执行的操作提前编码成命令列表，一次性批量提交给 GPU**，GPU 自己在后台异步执行，CPU 和 GPU 可以完全并行工作，大幅提升了执行效率。

![[Pasted image 20260413230524.png]]

#### 作用

`GPUCommandEncoder`是 CPU 用来**录制 GPU 指令**的工具，它会把你后续的所有渲染操作，转换成 GPU 可以识别的二进制命令，最终打包成命令缓冲区（`GPUCommandBuffer`）。

#### 背后原理

现代 CPU 和 GPU 是两个完全独立的并行处理器，两者通过**命令队列**通信：

- CPU 不能直接调用 GPU 的执行逻辑，只能把要 GPU 做的事情写成命令列表，一次性发给 GPU。
    
- 这种 “录制 - 提交” 的模式，把命令的生成和执行完全分离：CPU 在录制命令的时候，GPU 可以同时执行之前提交的命令，实现两者的并行工作，避免了传统 API 中 CPU 每调用一个函数就要等待 GPU 的开销。
    
- 同时，命令编码器支持**多线程并行录制**，你可以用多个 CPU 核心同时生成不同的命令，最后合并提交，这在多核 CPU 上可以大幅提升复杂场景的命令生成速度，这是 WebGL 无法做到的。

### 3.2 GPUCommandEncoder GPU命令解码器

有了前面的铺垫，我们也明白了为什么要创建`GPUCommandEncoder`。它是通过 `GPUDevice`创建的，而`GPUDevice` 最重要的方法之一是 `createCommandEncoder` ，它返回一个 `GPUCommandEncoder` 。这个对象负责创建和存储将要发送到GPU的命令。以下代码展示了如何创建它，如同`draw()`函数内的一样：
```typescript
const commandEncoder = device.createCommandEncoder();
```
创建了该对象之后， `GPUCommandEncoder` 的方法使得定义将被发送到设备进行处理的操作成为可能。表2.4列出了这些方法中的每一个。

一般来说，**前三个方法是最重要的**。也是我们刚才用到的方法。第一个方法 `beginRenderPass` 返回一个 `GPURenderPassEncoder` ，它将为GPU定义图形操作。完整探索 `GPURenderPassEncoder` 的能力将需要好几章的内容。

 GPUCommandEncoder 方法

|方法|描述|
|---|---|
|beginRenderPass()|开始编码一个渲染通道，用于执行图形渲染|
|beginComputePass()|开始编码一个计算通道，用于执行计算任务|
|finish()|返回一个包含编码器命令序列的命令缓冲区|
|clearBuffer()|编码一个命令，将一个缓冲区填充为零|
|copyBufferToBuffer()|编码一个命令，从一个缓冲区复制数据到另一个缓冲区|
|copyBufferToTexture()|编码一个命令，从一个缓冲区复制数据到一个纹理|
|copyTextureToBuffer()|编码一个命令，从一个纹理复制数据到一个缓冲区|
|copyTextureToTexture()|编码一个命令，从一个纹理复制数据到另一个纹理|
|insertDebugMarker()|在命令序列中插入一个标记，可用于调试|

表中的第二个方法 `beginComputePass()` 返回一个 `GPUComputePassEncoder` 。这个对象的字段存储了告诉GPU如何执行计算任务的信息。从第8章开始，我将解释如何使用它来执行与数学、机器学习和图像处理相关的操作。

最后一个方法 `finish()` 是在每一个非平凡的WebGPU应用程序中必需的。它将编码器的数据转换成一个包含一个或多个GPU命令的 `GPUCommandBuffer` 。一旦应用程序将这个缓冲区发送到设备的队列中，GPU就会开始执行这些命令。

### 3.3 渲染通道描述符 `GPURenderPassDescriptor`

创建`GPURenderPassDescriptor`的代码如下：

```typescript
const renderPassDescriptor: GPURenderPassDescriptor = {
  colorAttachments: [/* ... */]
};
```

在了解了渲染的背景和`CommandEncoder`之后，这个`RenderPass(渲染通道)`又是什么鬼，它难道就是被打包的命令？事实并非如此。

RenderPass（渲染通道）的官方定义是：

> 一组连续的、针对**同一组渲染附件**（也就是你要渲染到的目标纹理 / 缓冲区）的渲染操作的集合，它是现代 GPU 用来做批量优化的核心单元。

简单来说，我们可以把它理解成：**把一批 “都是画到同一个目标上” 的绘制操作，打包成一个整体，让 GPU 可以针对性的做优化**。
#### 作用

这是用来描述本次**渲染通道**的配置信息，提前告诉 GPU 这一组渲染操作的目标、输入输出的处理方式。

#### 核心配置的原理
```typescript
const renderPassDescriptor: GPURenderPassDescriptor = {
      // WebGPU API 的 GPURenderPassDescriptor 接口的 colorAttachments 属性是一个颜色附件数组。
      // 它用于描述渲染通道的颜色附件。
      // 它用于创建渲染通道编码器。
      colorAttachments: [
        {
          // clearColor 用于描述纹理将被清除为的颜色。
          clearValue: { r: 0.8, g: 0.8, b: 0.8, a: 1.0 },
          // loadOp 用于描述纹理将如何加载，在本例中，我们正在清空纹理
          loadOp: "clear",
          // toreOp 用于描述纹理的存储方式。在这种情况下，我们正在存储该纹理。
          storeOp: "store",
          // 视图用于描述将被渲染到的纹理。
          view: this.context.getCurrentTexture().createView()
        }
      ]
    };
```
其中的`colorAttachments`里的配置，是 GPU 优化的关键：

- **`clearValue`**：定义清除纹理的目标颜色，也就是我们看到的灰色背景。
    
- **`loadOp`****（加载操作）**：告诉 GPU，在渲染通道开始的时候，如何处理目标纹理原来的内容。这里用了`clear`，意思是直接把纹理清除成我们指定的颜色，GPU 就不需要从显存里读取纹理原来的内容了，直接覆盖，节省了大量的内存带宽。
    
- **`storeOp`****（存储操作）**：告诉 GPU，在渲染通道结束的时候，要不要把渲染的结果保存到目标纹理里。这里用了`store`，就是要保存，这样我们才能把最终的结果显示到画布上。
    
- **`view`**：定义本次渲染的目标纹理视图，也就是画布的后台纹理，告诉 GPU 我们要把渲染结果画到这里。
    

#### 背后原理

渲染通道（Render Pass）是现代 GPU 的核心优化单元： GPU 会把同一个通道内的所有渲染操作当成一个整体来优化：它提前知道了所有输入输出的配置，就可以提前做内存的预加载、状态的切换优化，甚至可以把多个渲染通道的操作重排，来提高执行效率。这和传统 API 中每次绘制都要切换状态、隐式处理纹理的方式比，效率高了很多。

## 层级关系：从录制到提交的完整流程

我用一个实际的 3D 渲染的例子，帮你理清楚这几个容易混淆的概念的层级：

比如你要渲染一帧完整的游戏画面，你会这么做：

1. **创建** **`CommandEncoder`**：这是你用来录制所有 GPU 命令的 “总录制工具”，相当于你用来写任务清单的纸和笔。
    
2. **开启第一个** **`RenderPass`** ：**渲染阴影贴图** 
    1. 这一组操作的目标是「阴影纹理」，你会把场景的深度信息渲染到这个纹理里，用来做后面的阴影计算。

    2. 这一步的所有绘制命令，都会被录制到这个 RenderPass 里。
        
    3. 完成后，结束这个 RenderPass。
        
3. **开启第二个** `RenderPass`：渲染主场景
    1. 这一组操作的目标是「画布纹理」，你会把主场景的模型、光照都画到画布上，过程中会用到刚才的阴影纹理。
        
    2. 这一步的所有绘制命令，都会被录制到这个新的 RenderPass 里。
        
    3. 完成后，结束这个 RenderPass。
        
4. **开启第三个** **`RenderPass`**：**渲染 UI**
    1. 这一组操作的目标还是「画布纹理」，你会在主场景的内容上，叠加 UI 元素。
        
    2. 这一步的所有绘制命令，都会被录制到这个新的 RenderPass 里。
        
    3. 完成后，结束这个 RenderPass。
        
5. **打包提交**：把 `CommandEncoder` 里录制的所有内容（也就是刚才的 3 个 RenderPass 的所有命令），打包成一个 `CommandBuffer`，一次性提交给 GPU。
    

你看，这三个 RenderPass，都是在同一个 `CommandBuffer` 里的，GPU 会按顺序执行它们，因为它们之间有依赖关系。

## 几个核心概念的对比

|                     |                                              |                             |
| ------------------- | -------------------------------------------- | --------------------------- |
| 概念                  | 作用                                           | 层级                          |
| `CommandEncoder`    | 用来录制所有 GPU 命令的总工具，相当于 “录音笔”                  | 最顶层的录制工具                    |
| `RenderPassEncoder` | 用来录制单个 RenderPass 内的渲染命令的子工具，相当于录音笔里的一个 “音轨” | CommandEncoder 的子分组         |
| `CommandBuffer`     | 最终录制好的所有命令的打包文件，也就是你提交给 GPU 的那个 “完整的录音”      | 最终的打包产物，里面可以包含多个 RenderPass |
| `RenderPass`        | 一组渲染操作的集合，对应 CommandBuffer 里的一个 “音轨”         | CommandBuffer 里的子分组         |

## 为什么需要把渲染操作分成 RenderPass？

核心原因是：**GPU 需要提前知道这一组操作都是针对同一个附件的，这样它才能做大量的针对性优化**，这也是现代 GPU 比旧版 GPU 快很多的核心原因之一：

### 1. 内存带宽优化（最核心的优化）

现代 GPU（尤其是移动 GPU）都有高速的**片元缓存（Tile Memory）**，它的速度比显存快几十倍。 如果 GPU 知道一组操作都是针对同一个附件，它就可以把这组操作的中间结果，都存在这个高速缓存里，不用每次绘制完都把结果写回慢得多的显存，直到整个 RenderPass 结束，再一次性把最终结果写回显存。 这可以节省 90% 以上的显存读写带宽，对移动设备的性能提升尤其明显。

### 2. 状态切换优化

GPU 知道同一个 RenderPass 里的所有操作，都是针对同一组附件的，所以它不需要在这些操作之间切换渲染目标，也不需要重新配置附件的格式、读写方式，减少了大量的状态切换开销。

### 3. 精细化的资源管理

通过 `loadOp` 和 `storeOp`，你可以告诉 GPU，在 RenderPass 的开始和结束，如何处理附件的内容，GPU 可以根据这个来做进一步的优化：

- 如果你用了 `loadOp: "clear"`，GPU 就知道，这个附件原来的内容没用了，不需要从显存里加载原来的内容，直接清除就好，节省了加载的开销。
    
- 如果你用了 `storeOp: "discard"`，GPU 就知道，这个 RenderPass 的结果不需要保存，结束的时候直接丢弃就好，不用写回显存。
    

### 3.4  GPUQueue

GPUDevice 有一个重要的属性，名为 queue ，它提供了对 GPUQueue 的访问。为了执行GPU操作，应用程序需要调用 GPUQueue 的方法，其中包括以下三个：

- writeBuffer ——更新GPU上缓冲区对象中的数据
- writeTexture ——更新GPU上纹理对象中的数据
- submit ——告诉GPU执行一个或多个命令

`submit` 方法尤为重要，因为它告诉GPU要执行哪些任务（图形渲染或通用计算）。具体来说， `submit` 接受之前讨论的 `GPUCommandEncoder` 创建的命令。一旦调用了这个方法，GPU就会开始执行缓冲区中的命令。
```typescript
// submit 用于向图形处理器（GPU）提交命令缓冲区。
// commandEncoder.finish () 用于创建命令缓冲区。
this.device.queue.submit([commandEncoder.finish()]);
```

注意我们前面说过，`GPUCommandEncoder`会把所有的`RenderPass`打包为一个`GPUCommandBuffer`。这里就是在`commandEncoder.finish()`这一步实现的。

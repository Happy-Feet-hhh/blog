## 1. 这一节我们要做什么？

在上一节里，我们已经拿到了这些关键对象：

- `GPUCanvasContext`
- `GPUAdapter`
- `GPUDevice`
- `GPUCommandEncoder`
- `GPURenderPassDescriptor`

并且我们已经能清屏了（把背景刷成灰色）。

这一节，我们要做的事情很明确：**在灰色背景上真正画出一个红色三角形（Hello Triangle）**。

这一步看似简单，但它把 WebGPU 最核心的一条链路串起来了：

1. 写 `WGSL` 着色器代码（GPU 要执行的程序）
2. 在 TypeScript 里创建 `ShaderModule`
3. 组合出 `GPURenderPipeline`（渲染管线状态对象）
4. 在 `RenderPass` 内绑定管线并发出 `draw(3)`
5. 提交命令给 GPU 执行

## 2. 先补上核心概念：图形渲染流水线到底是什么？

在真正进入代码前，先把图形渲染这件事从“流程层”讲清楚。

图形渲染的目标是：把 3D/2D 空间里的顶点数据，变成屏幕上的像素。这个过程不是一步完成，而是沿着一条流水线执行。现代 WebGPU / Vulkan / D3D12 都遵循类似模型。

![[image-3.png]]

上图可以理解成以下阶段：

1. **Input data**：输入顶点/索引/资源数据（位置、颜色、法线、纹理坐标等）
2. **Vertex shader**：对每个顶点执行一次，输出裁剪空间位置与可插值属性
3. **Assembly（图元组装）**：按拓扑规则把顶点拼成点/线/三角形
4. **Rasterization（光栅化）**：把图元覆盖区域离散成片元（fragment）
5. **Fragment shader**：对每个片元执行一次，决定输出颜色/深度等
6. **Color & blending operations**：做深度测试、模板测试、混合等
7. **Pixels**：最终写入颜色附件，显示到屏幕

### 2.1 可编程阶段 vs 固定功能阶段

|阶段|是否可编程|你在本章控制方式|
|---|---|---|
|Vertex Shader|是|WGSL `@vertex`|
|Primitive Assembly|否（固定规则可配置）|`primitive.topology`|
|Rasterization|否（固定规则可配置）|由管线状态控制|
|Fragment Shader|是|WGSL `@fragment`|
|Color/Blend/Depth|半固定（参数化）|`targets/depthStencil/blend` 等配置|

所以“渲染管线”这个词，不是单指 shader，而是把可编程和固定功能阶段一起描述的一整套执行规则。

## 3. 本节完整代码

### 3.1 `src/main.ts`

```typescript
// 从shader源代码引入
import shaderSource from "./shaders/shader.wgsl?raw";

class Renderer {
  private context!: GPUCanvasContext;
  private device!: GPUDevice;
  // 新增 渲染管线对象
  private pipeline!: GPURenderPipeline;

  constructor() {}
  
  public async initialize(): Promise<void> {
    if (!navigator.gpu) {
      alert("WebGPU不受支持!");
    }

    const canvas = document.getElementById('canvas') as HTMLCanvasElement;
    this.context = canvas.getContext('webgpu')!;
    
    if(!this.context) {
      alert("当前画布不支持WebGPU上下文!");
    }

    const adapter = await navigator.gpu.requestAdapter({
      powerPreference: "low-power",
    })!;

    if (!adapter) {
      alert("无法找到合适的适配器(显卡)")
    }

    const info = adapter?.info;
    console.log("显卡的厂商:", info?.vendor);
    console.log("显卡的架构:", info?.architecture);

    this.device = await adapter?.requestDevice()!;
    
    
    this.context.configure({
      device: this.device,
      format: navigator.gpu.getPreferredCanvasFormat(),
    });

    // 新增，准备shader module 着色器模块
    this.prepareModel();
  }

  // 新增 准备shaderModule 和 渲染管线
  private prepareModel(): void {
    const shaderModule = this.device.createShaderModule({
      code: shaderSource
    });

    const vertexState: GPUVertexState = {
      module: shaderModule,
      entryPoint: "vertexMain",
      buffers: []
    };

    const fragmentState: GPUFragmentState = {
      module: shaderModule,
      entryPoint: "fragmentMain",
      targets: [
        {
          format: navigator.gpu.getPreferredCanvasFormat()
        }
      ]
    };

    // 注意：WebGPU中不需要自己创建RenderPipeline的Layout, 因为可以auto
    // Render Pipeline Layout里保存了 bind_group相关的信息
    this.pipeline = this.device.createRenderPipeline({
      layout: "auto",
      vertex: vertexState,
      fragment: fragmentState,
      primitive: {
        topology: "triangle-list",
      }
    });
  }

  public draw() {
    const commandEncoder = this.device.createCommandEncoder();
    const rendePassDescriptor: GPURenderPassDescriptor = {
      colorAttachments: [
        {
          clearValue: { r: 0.8, g: 0.8, b: 0.8, a: 1.0 },
          loadOp: "clear",
          storeOp: "store",
          view: this.context.getCurrentTexture().createView()
        }
      ]
    };

    const passEncoder = commandEncoder.beginRenderPass(rendePassDescriptor);
    // DRAW HERE
    passEncoder.setPipeline(this.pipeline);
    passEncoder.draw(3); // draw 3 vertices

    passEncoder.end();

    const commandBuffer = commandEncoder.finish();
    this.device.queue.submit([commandBuffer]);
  }
}

const renderer = new Renderer();
renderer.initialize().then(() => renderer.draw());
```

### 3.2 `src/shaders/shader.wgsl`

```wgsl
struct VertexOut {
    @builtin(position) position: vec4f, // 顶点在裁剪空间中的位置
    @location(0) color: vec4f
}

// 顶点着色器会针对顶点数组中的每个顶点执行一次。
// 顶点索引通过内置变量传入。
@vertex
fn vertexMain(
    @builtin(vertex_index) vertexIndex: u32
) -> VertexOut {
    // 创建三角形的顶点数据，后期从cpu侧(ts)中引入
    let pos = array(
        vec2f( 0.0,  0.5),  // 上中点
        vec2f(-0.5, -0.5),  // 左下点
        vec2f( 0.5, -0.5)   // 右下点
    );

    var output: VertexOut;
    output.position = vec4f(pos[vertexIndex].x, pos[vertexIndex].y, 0.0, 1.0);
    output.color = vec4f(1.0, 0.0, 0.0, 1.0);

    return output;
}

// 片段着色器会针对三角形中的每个像素进行调用。
@fragment
fn fragmentMain(fragData: VertexOut) -> @location(0) vec4f {
    return fragData.color; // 像素的最终颜色
}
```

相较于上一节，新增逻辑可以概括成三件事：

也就是说，上一节解决的是“GPU 能收命令”；这一节解决的是“GPU 知道画什么、怎么画”。

## 5. Render Pass 概念与地位（先于代码理解）

很多人会把 `RenderPass`、`RenderPipeline`、`draw` 混在一起。这里先切清层级。

### 5.1 Render Pass 是什么？

`RenderPass`（渲染通道）是“一组针对同一组附件（attachment）的连续渲染操作”的执行上下文。

关键词是 **附件（attachment）**：

- 颜色附件（最终写颜色）
- 深度/模板附件（深度测试、模板测试）

你可以把它理解成：

- `RenderPass` 决定这段渲染“画到哪块目标内存 + 这块内存怎么开场/收尾”
- `RenderPipeline` 决定这段渲染“按什么规则画”

### 5.2 定义一个 RenderPass 需要哪些字段？

`beginRenderPass()` 传入 `GPURenderPassDescriptor`。常见字段如下：

|属性|是否必需|作用|
|---|---|---|
|`label`|否|调试标签|
|`colorAttachments`|是|颜色输出附件数组|
|`depthStencilAttachment`|否|深度/模板附件|
|`occlusionQuerySet`|否|遮挡查询|
|`timestampWrites`|否|GPU 时间戳写入|
|`maxDrawCount`|否|该 pass 最大 draw 次数提示|

本章只有 `colorAttachments` 必需，其他会在后续章节逐步启用。

### 5.3 `colorAttachments[i]` 每个字段的意义

|属性|是否必需|作用|
|---|---|---|
|`view`|是|当前附件对应的纹理视图|
|`resolveTarget`|否|多重采样解析目标|
|`loadOp`|是|pass 开始时如何处理旧内容（`load`/`clear`）|
|`storeOp`|是|pass 结束时是否保存结果（`store`/`discard`）|
|`clearValue`|条件必需|当 `loadOp: "clear"` 时的清屏值|

这也是为什么你的代码里：

```typescript
{
  clearValue: { r: 0.8, g: 0.8, b: 0.8, a: 1.0 },
  loadOp: "clear",
  storeOp: "store",
  view: this.context.getCurrentTexture().createView()
}
```

会得到“灰底 + 保存结果到画布”的效果。

## 6. TypeScript 代码逐段拆解

### 6.1 `import shaderSource from "./shaders/shader.wgsl?raw"`

这个导入有两个重点：

- `shader.wgsl` 是 GPU 程序源码文件；
- `?raw` 告诉 Vite：**把文件内容按字符串导入**。

所以 `shaderSource` 的实际类型是 `string`，刚好可以直接喂给：

```typescript
this.device.createShaderModule({ code: shaderSource })
```

如果没有 `?raw`，构建工具可能会把它当静态资源 URL，而不是源码字符串，`createShaderModule` 就没法用。

---

### 6.2 新增字段：`private pipeline!: GPURenderPipeline`

`GPURenderPipeline` 可以理解为 GPU 渲染阶段的一份“总配置单”，里面固定了：

- 顶点着色器入口函数是谁
- 片段着色器入口函数是谁
- 顶点输入布局是什么
- 光栅化拓扑是什么（点、线、三角形）
- 颜色输出格式是什么
- 绑定组布局（资源布局）是什么

注意：`pipeline` 一般是**初始化期创建、渲染期复用**，不是每帧都 new 一个。因为创建管线开销比较大，属于重对象。

---

### 6.3 `initialize()` 里新增 `this.prepareModel()`

顺序非常关键：

1. 先拿到 `device`
2. 再 `context.configure(...)`
3. 最后创建依赖 `device` 的 shader module 和 render pipeline

`prepareModel()` 依赖 `this.device`，所以必须在 `requestDevice()` 之后调用。

---

### 6.4 `prepareModel()`：创建 `ShaderModule`

```typescript
const shaderModule = this.device.createShaderModule({
  code: shaderSource
});
```

这一句的本质是：把 WGSL 文本交给驱动编译/验证，生成 GPU 可用的着色器模块对象。

它不等于“立即执行”，而是“准备好后续能被管线引用的程序块”。

---

### 6.5 `vertexState`：顶点阶段配置

```typescript
const vertexState: GPUVertexState = {
  module: shaderModule,
  entryPoint: "vertexMain",
  buffers: []
};
```

逐项看：

- `module`：顶点阶段使用哪个 `ShaderModule`
- `entryPoint`：该模块中哪个函数是顶点入口（对应 WGSL 的 `@vertex fn vertexMain(...)`）
- `buffers: []`：**没有绑定顶点缓冲区**

为什么 `buffers` 可以是空数组？因为本例顶点数据不是从 CPU 顶点缓冲传入，而是直接在 WGSL 里用数组 `pos` 写死，再通过 `@builtin(vertex_index)` 访问。

---

### 6.6 `fragmentState`：片段阶段配置

```typescript
const fragmentState: GPUFragmentState = {
  module: shaderModule,
  entryPoint: "fragmentMain",
  targets: [
    {
      format: navigator.gpu.getPreferredCanvasFormat()
    }
  ]
};
```

重点在 `targets[0].format`：

- 这代表片段着色器 `@location(0)` 的颜色输出要写进哪个像素格式的目标；
- 必须和画布上下文 `context.configure({ format })` 对齐。

你可以把它记成一条硬规则：**RenderPipeline 的 color target format 要和当前渲染目标纹理格式兼容**。

---

### 6.7 `createRenderPipeline(...)`：拼装渲染管线

```typescript
this.pipeline = this.device.createRenderPipeline({
  layout: "auto",
  vertex: vertexState,
  fragment: fragmentState,
  primitive: {
    topology: "triangle-list",
  }
});
```

几项关键参数：

- `layout: "auto"`：让 WebGPU 根据 shader 中的资源声明（`@group/@binding`）自动推导管线布局。本例里没有纹理/Uniform，所以它会推导出一个空布局。
- `vertex`：指定顶点阶段配置
- `fragment`：指定片段阶段配置
- `primitive.topology: "triangle-list"`：每 3 个顶点组成 1 个独立三角形

这也是为什么后面 `draw(3)` 就能刚好画一个三角形。

---

### 6.8 `draw()` 里的核心新增：`setPipeline + draw(3)`

```typescript
passEncoder.setPipeline(this.pipeline);
passEncoder.draw(3);
```

这是“真正画图”的起点。

- `setPipeline(...)`：告诉当前 RenderPass，后续 draw 命令用哪套着色器+固定功能配置
- `draw(3)`：提交一次非索引绘制，顶点数是 3

`draw(3)` 的简化版完整形态其实是：

```typescript
draw(vertexCount, instanceCount = 1, firstVertex = 0, firstInstance = 0)
```

所以本例等价于 `draw(3, 1, 0, 0)`。

GPU 会触发 3 次顶点着色器调用，`vertexIndex` 依次是 `0,1,2`，刚好取到 `pos` 数组的三个点。

## 7. GPURenderPipeline 概念、字段与图元拓扑

`GPURenderPipeline` 是“渲染规则总包”。`createRenderPipeline()` 的主字段建议先有全景认知：

|属性|是否必需|作用|
|---|---|---|
|`label`|否|调试标签|
|`layout`|是|资源绑定布局（`auto` 或显式 `GPUPipelineLayout`）|
|`vertex`|是|顶点阶段配置|
|`fragment`|否|片段阶段配置（只清屏可不设）|
|`depthStencil`|否|深度/模板测试配置|
|`primitive`|否|图元组装规则|
|`multisample`|否|多重采样设置|

### 7.1 为什么 `layout: "auto"` 能用？

因为本章 shader 里没有任何 `@group/@binding` 资源声明（没有 uniform/纹理/sampler），自动推导出的布局就是空布局，刚好成立。

### 7.2 `primitive` 字段详解

`primitive` 用来控制“顶点如何组装成图元”，核心是 `topology`。

|属性|作用|
|---|---|
|`topology`|组装类型：点、线、三角形及其 strip 变体|
|`stripIndexFormat`|strip 拓扑下索引格式（`uint16`/`uint32`）|
|`frontFace`|正面定义（`ccw`/`cw`）|
|`cullMode`|剔除模式（`none`/`front`/`back`）|
|`unclippedDepth`|是否禁用深度裁剪|

不同 `topology` 的直观形状如下：

这也是你当前配置 `triangle-list` 的直接含义：每连续 3 个顶点形成一个独立三角形。

## 8. WGSL 代码逐行解释（含标注语义）

### 8.1 `struct VertexOut`

```wgsl
struct VertexOut {
    @builtin(position) position: vec4f,
    @location(0) color: vec4f
}
```

这个结构体是“顶点阶段输出 / 片段阶段输入”的桥梁。

- `@builtin(position)`：
  - 这是固定语义，表示裁剪空间坐标（Clip Space Position）；
  - GPU 用它决定顶点落在屏幕哪里；
  - 类型必须是 `vec4f`（`x,y,z,w`）。
- `@location(0) color`：
  - 这是用户自定义插值通道 0；
  - 会从顶点阶段传到片段阶段；
  - 同一个 `location` 在上下游阶段必须类型兼容。

---

### 8.2 `@vertex fn vertexMain(@builtin(vertex_index) vertexIndex: u32)`

```wgsl
@vertex
fn vertexMain(
    @builtin(vertex_index) vertexIndex: u32
) -> VertexOut
```

几个关键标志：

- `@vertex`：声明这是“顶点着色器入口函数”
- `@builtin(vertex_index)`：读取当前是第几个顶点调用
  - 第一次调用是 `0`
  - 第二次调用是 `1`
  - 第三次调用是 `2`

它和 TS 的对应关系是：

- TS 里 `passEncoder.draw(3)` 决定会触发 3 次顶点调用；
- WGSL 里 `vertexIndex` 决定当前调用取哪个顶点数据。

---

### 8.3 顶点数组 `pos` 与裁剪空间坐标

```wgsl
let pos = array(
    vec2f( 0.0,  0.5),
    vec2f(-0.5, -0.5),
    vec2f( 0.5, -0.5)
);
```

这 3 个点在 NDC/裁剪空间中的直觉是：

- `(0.0, 0.5)`：上方中点
- `(-0.5, -0.5)`：左下
- `(0.5, -0.5)`：右下

组成一个正朝上的三角形。

坐标范围通常看作 `[-1, 1]`，超出会被裁剪（clip）。

---

### 8.4 `output.position = vec4f(..., 0.0, 1.0)`

```wgsl
output.position = vec4f(pos[vertexIndex].x, pos[vertexIndex].y, 0.0, 1.0);
```

为什么是 `vec4`：

- `x, y`：平面坐标
- `z`：深度（这里写 0）
- `w`：齐次坐标分量（通常写 1）

光栅化前 GPU 会做透视除法：`ndc = xyz / w`。本例 `w=1`，所以不改变坐标。

---

### 8.5 `output.color = vec4f(1.0, 0.0, 0.0, 1.0)`

这行给三个顶点都写了同样的红色，因此片段插值后的整个三角形仍然是红色。

如果三个顶点写不同颜色，你会看到颜色渐变（插值）。

---

### 8.6 `@fragment fn fragmentMain(...) -> @location(0) vec4f`

```wgsl
@fragment
fn fragmentMain(fragData: VertexOut) -> @location(0) vec4f {
    return fragData.color;
}
```

标志解释：

- `@fragment`：片段着色器入口
- 返回值 `@location(0)`：写入颜色附件 0

它和 TS 的对应关系是：

- TS 里 `fragment.targets[0]` 定义了颜色附件 0 的像素格式；
- WGSL 返回 `@location(0)` 对应写到这个 target。

## 9. 渲染管线与 RenderPass 的关系（核心）

这是最容易混淆的一组概念。先说结论：

- **RenderPipeline**：定义“怎么画”（着色器、顶点布局、拓扑、目标格式等固定状态）
- **RenderPass**：定义“画到哪里、这一段渲染如何装载和存储附件”

它们的关系是：**RenderPass 是执行上下文，Pipeline 是该上下文中的绘制规则模板**。

你可以在一个 RenderPass 里切换多个 Pipeline（例如先画网格，再画UI），也可以在一帧里开多个 RenderPass（例如先阴影，再主场景，再后处理）。

## 11. Shader 与 TypeScript 的一一映射

|WGSL|TypeScript 对应|作用|
|---|---|---|
|`@vertex fn vertexMain`|`vertex.entryPoint = "vertexMain"`|指定顶点阶段入口|
|`@fragment fn fragmentMain`|`fragment.entryPoint = "fragmentMain"`|指定片段阶段入口|
|`@builtin(vertex_index)`|`passEncoder.draw(3)` 触发 3 次调用|按顶点索引取数据|
|`@builtin(position)`|顶点输出必填语义|告诉 GPU 顶点屏幕位置|
|`@location(0) color`（顶点输出）|`fragment` 输入参数里的 `VertexOut`|跨阶段插值传递颜色|
|片段返回 `@location(0)`|`fragment.targets[0]`|写入颜色附件0|
|`vec4f(..., 1.0)`|无直接TS字段，但受光栅化规则约束|齐次坐标用于透视除法|

## 12. Draw 方法族：不止 `draw(3)`

虽然当前只用 `draw(3)`，但这一组 API 的分工建议一次理解清楚。

|方法|作用|
|---|---|
|`draw(...)`|按顺序读取顶点并绘制|
|`drawIndexed(...)`|按索引读取顶点并绘制|
|`drawIndirect(...)`|绘制参数来自 GPU 缓冲区|
|`drawIndexedIndirect(...)`|索引绘制 + 参数来自 GPU 缓冲区|

`draw` 的常见重载形式：

|调用形式|语义|
|---|---|
|`draw(vertexCount)`|绘制前 `vertexCount` 个顶点|
|`draw(vertexCount, instanceCount)`|额外做实例化渲染|
|`draw(vertexCount, instanceCount, firstVertex)`|从偏移 `firstVertex` 开始读|
|`draw(vertexCount, instanceCount, firstVertex, firstInstance)`|指定首实例编号|

你当前的 `draw(3)` 等价于 `draw(3, 1, 0, 0)`。

## 13. 缓冲区与布局：本章先建立“正确心智模型”

你现在这版 Hello Triangle 把顶点写死在 shader 里，是最小可运行方案；真实工程会把数据放在 GPUBuffer。

### 13.1 顶点属性数据长什么样？

![[image-5.png]]

上图展示了一个更典型的三角形顶点属性：每个顶点有位置 + 颜色。

例如交错布局（interleaved）通常写成：

```typescript
// [pos.x, pos.y, col.r, col.g, col.b] * 3 vertices
const vertexData = new Float32Array([
  0.0,  0.5, 0.0, 1.0, 0.0,
 -0.5, -0.5, 1.0, 0.0, 0.0,
  0.5, -0.5, 0.0, 0.0, 1.0,
]);
```

### 13.2 `createBuffer` 的关键字段

|属性|是否必需|作用|
|---|---|---|
|`label`|否|缓冲区调试名|
|`size`|是|字节大小|
|`usage`|是|缓冲区用途标志位|
|`mappedAtCreation`|否|创建时是否映射|

常见 `usage`：

|标志|说明|
|---|---|
|`GPUBufferUsage.VERTEX`|作为顶点缓冲|
|`GPUBufferUsage.INDEX`|作为索引缓冲|
|`GPUBufferUsage.UNIFORM`|作为 uniform 缓冲|
|`GPUBufferUsage.STORAGE`|作为 storage 缓冲|
|`GPUBufferUsage.COPY_DST`|可作为复制写入目标|
|`GPUBufferUsage.COPY_SRC`|可作为复制读取源|

### 13.3 顶点布局对象为什么重要？

GPU 不会“猜”哪些 float 是位置、哪些 float 是颜色，必须告诉它内存布局。

![[image-6.png]]

对应的布局对象（示意）：

```typescript
const bufferLayout: GPUVertexBufferLayout = {
  arrayStride: 20, // 每顶点 5 个 float * 4 字节
  attributes: [
    { shaderLocation: 0, offset: 0, format: "float32x2" }, // position
    { shaderLocation: 1, offset: 8, format: "float32x3" }, // color
  ],
};
```

关键字段解释：

|字段|含义|
|---|---|
|`arrayStride`|相邻两个顶点数据块的字节跨度|
|`offset`|当前属性在单个顶点数据块内的偏移|
|`format`|属性类型与分量数|
|`shaderLocation`|映射到 shader 输入位置|

这部分就是下一节从“硬编码顶点”升级到“顶点缓冲输入”的理论基础。

## 14. 两个容易踩坑的点

### 14.1 `context.configure().format` 与 `fragment.targets[0].format` 不一致

如果两边像素格式不匹配，常见表现是：

- 管线创建失败
- 或渲染输出异常

所以最稳妥的写法就是像本例一样，统一用：

```typescript
navigator.gpu.getPreferredCanvasFormat()
```

### 14.2 `buffers: []` 不是通用方案

本例可以空，因为顶点数据硬编码在 WGSL 里。真实项目通常会：

- 创建 `GPUBuffer` 存顶点数据
- 在 `vertex.buffers` 里声明步幅和 attribute 布局
- 在 RenderPass 里 `setVertexBuffer`

后续我们会切到这个更真实的路径。

## 16. 总结

到这里，我们完成了 WebGPU 教程中最关键的里程碑：**第一个真正可见的图元渲染**。

本节你应该掌握了这些核心点：

1. WGSL 不是“可选项”，而是 GPU 可执行逻辑本体
2. RenderPipeline 负责定义“如何渲染”
3. RenderPass 负责定义“渲染到哪、附件如何处理”
4. `draw(3)` + `@builtin(vertex_index)` 能在无顶点缓冲时完成最小三角形
5. `@location` / `@builtin` 是 WGSL 的语义连接点，必须和 TS 管线配置匹配
6. `primitive.topology` 决定顶点组装方式，不同模式会得到完全不同图元
7. 顶点缓冲与布局对象是工程化渲染的核心，不再依赖 shader 内联顶点

下一步，我们通常会做三件事：

- 把顶点数据从 shader 内联，迁移到 CPU 侧 `GPUBuffer`
- 引入 `uniform buffer`，让颜色/矩阵可动态更新
- 开始渲染循环（`requestAnimationFrame`）而不是只画一帧
